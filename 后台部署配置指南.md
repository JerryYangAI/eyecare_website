# 品川光医官网 · 内容后台部署配置指南

本指南帮助你把已写好的后台（Decap CMS）真正跑起来。代码我已经全部写好并推送到仓库，你需要做的是在 **GitHub** 和 **Cloudflare** 的网页后台里做几步配置（这几步涉及账号密钥，只能由你本人操作）。

全程约 **20–30 分钟**，不需要写代码。

---

## 一、它是怎么工作的（先看懂再动手）

```
你（编辑） →  打开 eyecare-website.pages.dev/admin
            │
            ├─ ① Cloudflare Access：先用邮箱收验证码登录（拦住所有非授权邮箱）
            │
            └─ ② Decap 后台：点“用 GitHub 登录”一次 → 编辑内容 → 保存
                        │
                        └─ 自动提交到 GitHub 仓库 → Cloudflare Pages 自动重新部署 → 前端更新
```

- **登录门禁** = Cloudflare Access（你要的“邮箱验证码”方式）。只有你白名单里的邮箱能打开后台。
- **保存内容** = Decap 通过 GitHub 提交。需要一次 GitHub 授权（令牌会被浏览器记住，不用每次登）。
- 前端每个版块都对应后台一个模块：**网站设置 / 文案管理 / 图片管理 / 视频管理 / 资质证书 / 文献资料**。后台一改、保存即提交，网站几十秒后自动更新。

> 说明：邮箱验证码（Access）负责“谁能进后台”，GitHub 授权负责“把改动写回网站”。两者各司其职，是这类纯静态网站最稳的做法。

---

## 二、第 1 步：建 GitHub OAuth App（约 5 分钟）

用于让后台能把改动提交回仓库。

1. 打开 https://github.com/settings/developers → **OAuth Apps** → **New OAuth App**
2. 填写：
   - **Application name**：`KOUSHICARE CMS`
   - **Homepage URL**：`https://eyecare-website.pages.dev`
   - **Authorization callback URL**：`https://koushi-auth.<你的子域>.workers.dev/callback`
     （这个 Worker 域名在第 2 步创建后才有；可先随便填，建好 Worker 后回来改成真实地址）
3. 点 **Register application**
4. 记下 **Client ID**；再点 **Generate a new client secret**，记下 **Client Secret**（只显示一次）

---

## 三、第 2 步：部署 OAuth 代理 Worker（约 8 分钟）

代码在仓库 / 项目文件夹的 `官网方案/cms-auth/worker.js`。两种部署方式任选其一。

### 方式 A：网页控制台（推荐，无需命令行）
1. 进 Cloudflare 控制台 → **Workers & Pages** → **Create** → **Create Worker**
2. 名称填 `koushi-auth`，创建后点 **Edit code**
3. 把 `worker.js` 的全部内容粘贴进去，**Deploy**
4. 部署后会得到域名，例如 `https://koushi-auth.你的子域.workers.dev`
5. 进该 Worker 的 **Settings → Variables and Secrets**，添加两个 **Secret**：
   - `GITHUB_CLIENT_ID` = 第 1 步的 Client ID
   - `GITHUB_CLIENT_SECRET` = 第 1 步的 Client Secret
6. 回到第 1 步的 GitHub OAuth App，把 **Authorization callback URL** 改成
   `https://koushi-auth.你的子域.workers.dev/callback`（确保和真实 Worker 域名一致）

### 方式 B：命令行（wrangler）
```bash
cd 官网方案/cms-auth
npm i -g wrangler
wrangler login
wrangler deploy
wrangler secret put GITHUB_CLIENT_ID       # 粘贴 Client ID
wrangler secret put GITHUB_CLIENT_SECRET   # 粘贴 Client Secret
```

## 四、第 3 步：把后台指向你的 Worker（约 2 分钟）

编辑仓库里的 `admin/config.yml`，把这一行：

```yaml
    base_url: https://koushi-auth.example.workers.dev   # TODO
```

改成你真实的 Worker 域名：

```yaml
    base_url: https://koushi-auth.你的子域.workers.dev
```

> 可以直接在 GitHub 网页上点 `admin/config.yml` → 铅笔图标编辑 → Commit。提交后 Pages 会自动重新部署。

---

## 五、第 4 步：用 Cloudflare Access 设“邮箱验证码”门禁（约 8 分钟）

这一步实现你要的“**通过邮箱发送验证码进入**”。

1. Cloudflare 控制台 → **Zero Trust**（首次进入会让你起一个团队名，如 `koushi`，免费档够用）
2. 左侧 **Access → Applications → Add an application → Self-hosted**
3. 配置：
   - **Application name**：`KOUSHICARE Admin`
   - **Session Duration**：建议 `24h`
   - **Application domain**：
     - 域名：`eyecare-website.pages.dev`
     - 路径：`admin`（即只保护 `/admin` 这个后台路径，不影响公开网站）
   - 建议再加一条同样策略保护 `admin/config.yml`（可选）
4. **下一步 Policies** → Add a policy：
   - Policy name：`Allowed editors`
   - Action：`Allow`
   - **Include → Emails** → 填你的邮箱（可加多个团队成员邮箱）
5. 保存。**登录方式默认就是 One-time PIN（一次性邮箱验证码）** —— 访客输入邮箱，Cloudflare 发 6 位验证码到邮箱，输入即进。无需额外配置。
6. 完成后访问 `https://eyecare-website.pages.dev/admin`，会先跳到 Cloudflare 的邮箱验证码页面。

> 如果在 Zero Trust 的 **Settings → Authentication** 里看到 One-time PIN 没启用，手动启用它即可。

---

## 六、第 5 步：验收测试（约 3 分钟）

1. 打开 `https://eyecare-website.pages.dev/admin`
2. 应先出现 **Cloudflare 邮箱验证码**页面 → 输入你白名单里的邮箱 → 收验证码 → 输入进入
3. 进入后台后点 **“Login with GitHub”**，授权一次
4. 左侧能看到六个模块：网站设置 / 文案管理 / 图片管理 / 视频管理 / 资质证书 / 文献资料
5. 随便改一句文案（如“网站设置 → 首屏 Slogan”）→ **Save** → **Publish**
6. 等约 30–60 秒（Pages 重新部署），刷新网站首页，确认改动已生效

> 用一个**不在白名单**的邮箱测试，应被 Cloudflare 拦在验证码页之外——说明门禁生效。

## 七、各模块对应关系速查

| 后台模块 | 对应前端版块 | 编辑的文件 |
|---|---|---|
| 网站设置 | SEO（标题/介绍/关键词）+ 首屏文案 | `content/site.json` |
| 文案管理 | 问题共鸣 / 科学原理 / 为什么是红光 / 技术溯源 / 研发与生产 / 资质与荣誉 / 用户反馈 / 导航 / 底端 | `content/copy.json` |
| 图片管理 | 首屏图、问题共鸣四图、泪膜图、研发/生产/检测图 | `content/images.json` |
| 视频管理 | 红光、热敷、工厂实景、用户反馈视频 | `content/videos.json` |
| 资质证书 | 首屏证书墙（可增删、拖动排序） | `content/certs.json` |
| 文献资料 | 文献原文 PDF（标题+年份+上传文件） | `content/refs.json` |

媒体文件存放：图片 → 仓库 `images/uploads/`；视频 → 仓库 `videos/`；PDF → 仓库 `files/`。全部随网站一起部署（你选择的“全部放 GitHub 仓库”方案）。

## 八、常见问题

- **后台空白/转圈**：多半是 `base_url` 没改成真实 Worker 域名，或 Worker 的两个 Secret 没配好。
- **点 GitHub 登录后报错**：检查 GitHub OAuth App 的 callback URL 是否与 Worker 域名完全一致（含 `/callback`）。
- **改了不生效**：确认点了 **Publish**（不是只 Save 草稿）；并等 Pages 部署完成。
- **“图片绑定位置”下拉**：请勿改动，它决定图片显示在前端哪个位置；只换图片文件即可。
- **想加协作者**：在第 4 步的 Access Policy 里把对方邮箱加进 Include。
